# Wire Bonder Intake Checklist

这份清单用于向软件部申请第一轮只读试点输入。目标是先做小范围验证，不接入核心仓库，不修改源码，不生成 patch。

## 可直接发送的请求

我想申请一个非核心模块或历史版本的 wire bonder 只读代码目录，用来做本地代码知识库试点。第一步只做结构扫描和小范围检索验证，不上传代码、不训练模型、不修改源码、不写 SVN。

请协助提供以下内容：

1. 一个只读代码目录，建议不超过 5 万行。
2. 目录范围说明：这个目录大致属于哪个模块，是否包含第三方库或 generated code。
3. 编码说明：是否存在 GBK / GB18030 / 混合编码。
4. 10 个真实工程问题，尽量覆盖入口、错误、状态机/流程、回零、报警码/命令码、模块边界。
5. 一位熟悉该模块的工程师，帮忙确认返回的 file:line 是否正确。

如果第一轮效果不好，我们会停止在 probe 报告和小范围索引，不影响现有开发流程。

## 输入清单

| 项目 | 必需 | 说明 | 示例 |
|------|------|------|------|
| 只读代码目录 | 是 | 非核心、历史版本或可脱敏目录；不需要能编译 | `D:/WireBonder/Pilot/Motion` |
| 目录范围说明 | 是 | 模块职责、是否含第三方库、是否含生成代码 | 运控模块，不含视觉，不含第三方 |
| 编码说明 | 是 | UTF-8 / GBK / GB18030 / 混合编码 | 老文件可能是 GBK |
| 10 个真实问题 | 是 | 用工程师日常会问的定位问题 | 报警码在哪里定义和处理 |
| 验收人 | 是 | 至少一位工程师确认答案是否靠谱 | 模块 owner 或资深维护者 |
| 保密边界 | 是 | 是否允许本地 LLM，是否必须完全离线 | 只能内网，不能外发 |
| 排除目录 | 可选 | 明确哪些目录不要扫 | `third_party/`, `generated/`, `bin/` |
| 关键词表 | 可选 | 命令码、报警码、菜单 ID、模块简称 | `ERR_AXIS_TIMEOUT`, `CMD_HOME` |

## 10 个真实问题的格式

每个问题尽量包含：

- 问题文本。
- 问题类别：入口定位、错误追踪、状态机/流程、回零、报警码/命令码、模块边界、配置加载、历史遗留。
- 已知关键词：报警码、命令码、类名、日志文本、界面文字或配置项。
- 验收依据：工程师预期应该出现的文件、函数或模块。

推荐先填 `eval/wirebonder_questions_template.json`，其中 `ground_truth` 可先保持 `null`，等工程师确认后再补。

## 保密和安全边界

- 代码只在本地或内网机器处理。
- `kb probe` 只读取源码并生成本地报告。
- `kb index build` 只生成本地索引，不写回源码目录。
- 不修改源码，不生成 patch，不写 SVN。
- LLM 只看检索命中的 chunk 片段，不看全仓库全文。
- 可切换离线模型，做到完全不外发。
- 查询日志和错误日志保存在本地。
- 索引按版本目录保存，可随时删除或回滚。

## 拒绝或暂缓条件

以下情况建议暂缓，不要硬推试点：

- 只能提供核心生产分支，且没有只读副本。
- 无法确认代码外发边界。
- 没有工程师能验收 10 个问题。
- 目录包含大量二进制、生成代码或第三方库，且不能排除。
- 试点目标变成“自动修 bug / 自动提交 SVN”。这不属于当前阶段。
