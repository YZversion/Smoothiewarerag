# Stakeholder Pitch

## 可直接发送的软件部话术

我想先做一个非常小范围、只读、可回滚的代码知识库试点，用来验证它能不能帮助定位 wire bonder 代码里的入口、错误处理、状态流转和报警码。

第一步不需要接入核心仓库，也不会修改任何代码。只需要提供一个非核心模块或历史版本的只读目录，建议不超过 5 万行。我会先在本地运行结构扫描，输出一份报告，里面只有文件统计、编码统计、ctags 符号统计、超长文件和 generated code 风险，不会上传代码，也不会训练模型。

如果这一步报告没有明显风险，再本地构建索引，并请工程师给 10 个真实问题做验收，例如：

- 运动命令从界面下发后，进入运控模块的入口在哪里？
- 某个报警码在哪里定义、在哪里抛出、哪里处理？
- 回零流程的主要调用链是什么？
- 设备状态机有哪些状态，切换条件在哪里？
- 视觉模块和运控模块之间怎么通信？

验收时我只看工具能否返回相关源码、函数名和 file:line 引用。它不会自动改 bug，不会提交 SVN，不会重构代码，也不会保证答案 100% 正确。所有结论都必须能回到源码行号核查。

如果 10 个问题里多数能定位到正确代码，再讨论是否扩大范围；如果效果不好，就停止或只保留扫描报告，不影响现有开发流程。

## 需要对方提供什么

| 项目 | 要求 |
|------|------|
| 只读代码目录 | 非核心模块、历史版本或可脱敏目录均可；第一轮建议不超过 5 万行 |
| 编码说明 | 是否有 GBK / GB18030 / 混合编码 |
| 构建信息可选 | 不需要能编译；只需源码目录 |
| 10 个真实问题 | 覆盖入口、错误、状态机、回零、报警码、模块边界 |
| 人工验收人 | 至少一位熟悉该模块的工程师确认返回结果是否靠谱 |

## 5 步试点边界

1. 提供一个非核心只读目录。
2. 运行 `kb probe`，只输出结构报告。
3. 确认编码、ctags、超长文件、generated code 风险。
4. 本地构建索引，用 10 个问题验收。
5. 工程师评分后决定是否扩大范围。

## 风险控制回答

**会不会上传代码？**  
不会。扫描、索引、日志和报告都在本地或内网机器。

**LLM 会不会看到整个仓库？**  
不会。只有检索命中的少量 chunk 会进入 prompt；也可以使用离线模型。

**会不会改 SVN？**  
不会。当前阶段只读，不生成 patch，不提交 SVN。

**如果索引质量不好怎么办？**  
停止试点或只保留 probe 报告。索引目录可删除，源码目录不受影响。

**为什么先要真实问题？**  
因为工具价值不能靠 Smoothieware 的 demo 证明。wire bonder 的入口、报警、状态机和模块边界必须用工程师真实问题验收。
