# First Pilot Acceptance

这份文档定义 wire bonder 第一轮只读试点如何判定成功、失败或暂缓。它只评价 Level 0 代码问答和定位能力，不评价自动修改代码能力。

## 试点范围

| 项目 | 范围 |
|------|------|
| 代码范围 | 一个非核心只读目录，建议不超过 5 万行 |
| 问题数量 | 10 个真实工程问题 |
| 输出形式 | 源码文件、函数或类、file:line 引用、简短解释 |
| 操作边界 | 不修改源码，不生成 patch，不写 SVN |
| 数据边界 | 代码不外发；LLM 只看检索片段；可完全离线 |

## 成功标准

首轮试点判定为成功，需要同时满足：

1. `kb probe` 能生成完整报告，包含文件统计、编码统计、ctags 统计、风险文件和索引可行性评估。
2. `kb index build` 能完成，`kb index check` 通过。
3. 10 个真实问题中，至少 7 个能返回工程师认可的可核查 `file:line`。
4. 返回结果中不能出现无法追溯到源码 chunk 的关键事实。
5. 查询过程不修改源码目录，不写 SVN，不上传全仓库源码。
6. 工程师愿意基于结果继续补充问题或扩大到第二个目录。

## 部分成功标准

满足以下条件时，判定为部分成功，建议修规则后再试一轮：

- 10 个真实问题中有 4 到 6 个返回可核查 `file:line`。
- 失败问题集中在同一类，如报警码、消息 ID、MFC 宏或动态分发。
- probe 报告显示风险可控，可以通过排除目录、补 dispatch 规则或补 hint group 改善。
- 工程师认为输出虽不完整，但能节省读代码时间。

## 失败标准

出现以下任一情况，判定为失败或暂停：

- 10 个真实问题中少于 4 个能返回可核查 `file:line`。
- 多数答案无法回到源码核查，或引用与解释明显不一致。
- probe 显示编码、ctags、超长文件或 generated code 风险过高，且无法排除。
- 索引构建频繁失败，无法稳定复现。
- 需要上传核心源码到外部服务才能继续。
- 试点被要求自动修改代码、生成 patch 或提交 SVN。

## 人工评分表

每个问题由工程师按 0-3 分评分：

| 分数 | 含义 |
|------|------|
| 3 | 命中核心文件和函数，file:line 可核查，解释基本正确 |
| 2 | 命中相关文件，但缺少关键函数或链路不完整 |
| 1 | 有弱相关结果，但不能直接帮助定位 |
| 0 | 错误、不相关、无可用引用 |

建议通过线：

- 10 题总分 >= 21。
- 且至少 7 题评分 >= 2。
- 且没有安全边界违规。

## 必须记录的证据

每次试点至少保留：

- `reports/wire_bonder_probe.md`
- `reports/wire_bonder_probe.json`
- `data/index_vYYYYMMDD/index_manifest.json`
- 10 个问题的原文和类别
- 每题 top hits、工程师评分、备注
- 是否调用 LLM；如果调用，确认只发送检索 chunk

## 回滚策略

- 索引输出到独立版本目录，例如 `data/index_v20260630/`。
- `kb index check` 通过后再作为当前索引使用。
- 旧索引保留，不覆盖。
- 失败时删除新索引目录即可；源码目录不受影响。
- 所有报告和日志保存在本地，按公司保密要求归档或删除。

## 下一步判定

| 结果 | 下一步 |
|------|--------|
| 成功 | 收集第二批问题，扩大到相邻非核心目录 |
| 部分成功 | 分析失败类别，补充 dispatch / hint / 排除目录，再复测 |
| 失败 | 停止功能扩展，只保留 probe 报告和风险结论 |

## 明确不进入的工作

- 不做 Phase E edit-plan。
- 不生成 patch / diff。
- 不自动修 bug。
- 不提交 SVN / Git。
- 不上向量库、Agent、LangChain 或知识图谱主流程。
