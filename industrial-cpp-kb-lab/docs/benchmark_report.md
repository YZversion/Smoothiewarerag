## 二、索引构建耗时（B.2）

测试命令：
```
kb index build --repo-root repos/scale_test --src-root repos/scale_test
```

| 阶段 | 耗时 | 输出大小 | 备注 |
|------|------|----------|------|
| 01 scan | 1.0 s | 125 KB（895 条） | rg 扫描目录 |
| 02 symbols（ctags） | **24.0 s** | 12,465 KB | **瓶颈：ctags 串行解析** |
| 03 chunks | 0.6 s | 14,209 KB（7,347 chunks） | 符号边界切分 |
| 03 callgraph | 0.5 s | 3,917 KB | rg 提取调用关系 |
| 05 dispatch | 0.9 s | 0 KB | G-code 专用，scale_test 无相关代码 |
| **Total** | **27.0 s** | — | — |

对比 Smoothieware（18 万行）：约 8 s 完成全部阶段。

**ctags 是构建阶段唯一瓶颈**（占 89% 时间），且线性增长（18万→80万 ≈ ×4.4，耗时 ×3）。

### 已知问题

- `dispatch_index.json` 仅对 Smoothieware 有效（G-code dispatch 模式），其他项目输出为空
- 超大单文件：`raylib/src/external/miniaudio.h`（95,844 行）、`RGFW.h`（16,103 行）——这类文件会显著拖慢 rg 检索

## 四、瓶颈分析

### 检索层架构

每次 `kb search` 调用以下子流程：

```
BM25 (in-memory) → rg full-scan → symbol lookup → callgraph → merge & rank
```

BM25 本身快（< 5ms），**rg full-scan 是主要瓶颈**：它在所有文件上全量正则匹配，时间与 `文件数 × 平均文件大小` 成正比。

### 各因素贡献

| 因素 | 影响 | 说明 |
|------|------|------|
| rg 全量扫描 | **主要** | 895 文件 × 平均 900 行 = 80 万行逐行扫描 |
| 超大单文件 | 重要 | miniaudio.h（96k 行）单文件扫描 ~200ms |
| 查询 tokens 数量 | 中等 | token 越多 rg 越慢（多次调用） |
| BM25 检索 | 极小 | 7,347 chunks 内存检索 <5ms |
| symbol lookup | 极小 | hash table O(1) |

### Smoothieware P95 为何也未达标？

即使只有 269 个文件（18 万行），P95 也达 2.4 秒。原因是 rg 每次查询都全量扫描，加上网络/IO 抖动和 rg 进程启动开销。这不是规模问题，**是架构问题**。

## 六、缓解方案

### 短期（Phase B 后可做，不影响当前功能）

1. **排除超大单文件**：在 `01_scan_files.py` 里跳过 > 5000 行的文件（`miniaudio.h`、`dr_flac.h` 等 embedded 第三方库无需检索）；预计 P95 降低 30–50%

2. **rg 候选集限流**：先用 BM25 找 top-k 文件，再只对这些文件跑 rg，而非全量扫描；预计 P95 降低 60–80%

3. **rg 并发 timeout**：给 rg subprocess 加 `timeout=2s`，超时直接跳过返回 BM25 结果

### 中期（Phase C/D 时统一做）

4. **索引文件路径缓存**：将 BM25 命中文件列表缓存，同一 session 内相似查询复用

5. **ctags 并行化**：构建阶段 ctags 串行处理 895 文件耗时 24s，改为多进程可降至 ~6s

### 长期

6. **向量检索替换 rg**：用嵌入模型做 ANN 检索，摆脱 O(n×文件大小) 依赖；需 GPU 或 CPU ONNX 运行时

## 七、结论

| 结论 | 说明 |
|------|------|
| **索引构建可接受** | 80 万行 27s，生产可用（一次性任务） |
| **无崩溃** | 895 文件全部成功，ctags 失败率 0.5%（5/895） |
| **P95 未达标** | 12.6s >> 500ms；即使 18 万行也是 2.4s >> 500ms |
| **根因：rg 全扫** | 不是规模问题，是检索架构问题 |
| **最快修复** | 排除超大文件 + rg 候选集限流；无需改架构 |
| **系统上限（现架构）** | 约 5 万行以内可接近 500ms；更大规模需要限流或换检索方式 |

**建议：** Phase C 开始前，先做"排除超大文件"优化（1-2 小时），把 Smoothieware P95 降到 500ms 以内再继续。
